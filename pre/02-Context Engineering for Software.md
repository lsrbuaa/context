# 02-Context Engineering for Software

【技术博客】

## 摘要

本文用于展示 Context Engineering 在软件工程中的应用：让 AI 编程助手理解代码库、任务意图、约束条件、测试反馈和变更边界，从而更稳定地完成开发、调试和重构任务。

## 背景

软件工程任务天然依赖上下文。一个看似简单的代码修改，往往需要理解模块边界、接口契约、历史实现、测试结构、构建系统和部署环境。

如果上下文缺失，模型容易给出看似合理但无法落地的修改；如果上下文过载，模型又可能被无关文件干扰。因此，软件场景尤其需要系统化的上下文组织方法。

## 核心上下文

- Repository Map：目录结构、关键模块、依赖关系。
- Task Context：用户目标、验收标准、非目标范围。
- Code Context：相关源码、接口定义、调用链。
- Runtime Context：错误日志、测试输出、构建配置。
- Change Context：已有工作区变更、代码风格、兼容性要求。

## 实践流程

1. 先读代码库结构，不直接假设实现方式。
2. 用搜索定位相关模块、测试和接口。
3. 建立最小充分上下文，只读取与任务相关的文件。
4. 小范围修改，保持与现有模式一致。
5. 运行测试或构建命令验证结果。
6. 输出变更说明、验证结果和剩余风险。

## 示例场景

- 修复一个跨模块 bug。
- 为已有 API 增加一个字段。
- 重构重复逻辑但保持行为兼容。
- 根据测试失败信息定位根因。

## 展示要点

- 代码代理的关键能力不是“生成代码”，而是“维护正确上下文”。
- 软件上下文应服务于变更边界和验证闭环。
- 好的上下文工程可以显著降低幻觉式修改和无关重构。

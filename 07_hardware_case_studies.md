# Phase 7：典型智能硬件案例研究

## 1. 阶段研究目标

选取四类典型智能硬件形态，按统一的完整链路分析 Context 管理机制，验证 Phase 4-6 提出的架构方案在真实设备场景中的适用性。

每类设备均按以下链路拆解：
```
感知输入 → Context 化 → 记忆写入 → 检索召回 → Context 组装 → 模型决策 → 设备输出 → 用户反馈 → 记忆更新
```

## 2. 核心问题清单

1. 每类设备的核心感知模态和 context 来源是什么？
2. 事件切分在各设备形态上的特异性？
3. 各设备最需要的记忆层（L4-L6）侧重点？
4. Context Compiler 在各设备上的 Ranker 权重如何调整？
5. 各设备的主动服务场景和触发机制？
6. 各设备独有的隐私和安全挑战？
7. 实际产品中的已知问题对 Context 管理的启示？

### 案例 1：AI 眼镜

#### 设备定位
视觉感知型个人助手，通过空间理解和主动提醒增强用户的日常认知。

#### 完整 Context 链路

```
┌─ 感知输入 ─────────────────────────────────────────────────────┐
│  摄像头(30fps) + 麦克风 + IMU + GPS/Wi-Fi                       │
│  → 场景理解: VLM 描述当前视野                                   │
│  → 人物识别: 人脸匹配已知联系人                                 │
│  → 语音: ASR + 说话人分离                                       │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ Context 化 ──────────────────────────────────────────────────┐
│  视觉帧 → 场景描述文本 (每 2-5 秒更新)                         │
│  音频流 → 对话转录 (实时)                                       │
│  IMU → 活动标签 (walking/sitting/head_turn)                    │
│  GPS → 语义地点 (office_3F / home / commute)                   │
│  切分触发: 场景变化 / 人物变化 / 活动变化                       │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 记忆写入 ────────────────────────────────────────────────────┐
│  L2: 当前视觉场景 + 人物 + 活动 (持续更新)                     │
│  L4: 重要事件写入 (importance > 0.6)                           │
│      例: "14:30 在3F会议室与张工讨论API截止日期"                │
│  L5: 人物画像更新 / 地点知识更新                               │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 检索召回 ────────────────────────────────────────────────────┐
│  触发: 识别到已知人物 / 到达已知地点 / 用户注视时长>2s          │
│  检索: SpatialRelevance↑ + 人物关联记忆 + 近期事件             │
│  示例: 识别到张工 → 召回"上周他提到API延期风险"                 │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ Context 组装 (Compiler) ─────────────────────────────────────┐
│  Budget: ~4K tokens (VLM) 或 ~8K tokens (LLM)                 │
│  结构: System + User + Scene(重) + Memories + Task + Safety    │
│  Ranker 侧重: SpatialRelevance(0.25), Confidence(0.15)        │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 模型决策 ────────────────────────────────────────────────────┐
│  判断: 是否需要提醒用户？提醒内容是什么？                       │
│  考虑: 用户当前 interruptibility + 信息紧急程度                 │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 设备输出 ────────────────────────────────────────────────────┐
│  AR 叠加: 文字提示 (1行，3-5秒后消失)                          │
│  语音: 低声提示 (仅在独处/安全时)                              │
│  震动: 轻微震动 (仅作为注意力引导)                             │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 反馈 → 记忆更新 ────────────────────────────────────────────┐
│  用户看了提示 → 确认有用 → 强化该记忆                          │
│  用户忽略提示 → 可能不需要 → 提高触发阈值                     │
│  用户关闭提示 → 明确不需要 → 记录负反馈                       │
└───────────────────────────────────────────────────────────────┘
```

#### 独有挑战

| 挑战 | Context 管理策略 |
|------|-----------------|
| 拍摄旁人隐私 | 端侧人脸检测→未知人物模糊化→不存储原始图像 |
| 视觉 token 开销大 | 场景描述用自然语言而非原始图像（压缩比 ~1000:1） |
| 社交场合不能大声输出 | Compiler 检查 interruptibility + 输出通道约束 |
| 视觉注意力误判 | 用户注视≠用户关注，需要多模态融合(注视+头部朝向+对话context) |

#### 参考产品：Meta Ray-Ban AI [S039]
- 支持第一人称拍照/视频 + AI 视觉问答
- 局限：无 AR 显示、无主动提醒、记忆能力有限
- Context 启示：当前产品仅支持"用户主动拍照后问模型"，尚未实现持续感知+主动服务

### 案例 2：AI 耳机 / 耳戴设备

#### 设备定位
对话增强型助手，通过实时语音理解和低打扰提示辅助用户社交和工作。

#### 完整 Context 链路

```
┌─ 感知输入 ─────────────────────────────────────────────────────┐
│  麦克风(双/多麦阵列) + IMU(头部动作) + 蓝牙(附近设备)          │
│  → VAD: 检测语音活动                                           │
│  → ASR: 实时语音转文字                                         │
│  → 说话人分离: 区分用户/对方/多人                              │
│  → 声纹匹配: 识别已知联系人                                    │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ Context 化 ──────────────────────────────────────────────────┐
│  音频流 → 对话转录 (实时, 按说话人分段)                         │
│  声纹 → 说话人标签 (匹配已知联系人或标记 unknown)               │
│  头部 IMU → 用户注意力方向                                      │
│  切分触发: 对话主题变化 / 说话人变化 / 静默>30s                 │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 记忆写入 ────────────────────────────────────────────────────┐
│  L2: 当前对话状态 (话题/说话人/情绪) 持续更新                  │
│  L4: 对话事件 (有实质内容时): 话题、参与者、要点               │
│  L5: 人物画像(更新声纹/话题关联)、话题知识                     │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 检索召回 ────────────────────────────────────────────────────┐
│  触发: 对方提到用户可能需要帮助的内容 + 用户短暂停顿           │
│  检索: Recency↑ + 人物关联 + 话题匹配                          │
│  示例: 对方问"Alpha项目进度" → 召回"上周五完成第一阶段"        │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ Context 组装 (Compiler) ─────────────────────────────────────┐
│  Budget: ~2K tokens (极精简)                                   │
│  结构: System + 对话context + 检索记忆 + 输出约束              │
│  Ranker 侧重: Recency(0.25), TaskRelevance(0.30)              │
│  特殊约束: 输出必须 ≤3 词                                      │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 模型决策 ────────────────────────────────────────────────────┐
│  判断: 用户是否需要帮助? 介入的社交风险多大?                    │
│  ShouldIntervene = NeedProb × InfoValue × (1-SocialRisk)       │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 设备输出 ────────────────────────────────────────────────────┐
│  低声语音: 1-3 词 (如"上周五完成")                             │
│  延迟要求: <300ms (必须在对话语境窗口内)                       │
│  可忽略性: 用户可以不理会，不影响当前对话                       │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 反馈 → 记忆更新 ────────────────────────────────────────────┐
│  用户使用了提示信息 → 强化该记忆 + 确认提示策略                │
│  用户自行回答(未用提示) → 中性反馈                             │
│  用户被打断/尴尬 → 强负反馈 → 提高 SocialRisk 阈值            │
└───────────────────────────────────────────────────────────────┘
```

#### 独有挑战

| 挑战 | Context 管理策略 |
|------|-----------------|
| 极短输出约束 (1-3 词) | Compiler 的 output 指令必须明确限制长度 |
| 超低延迟 (<300ms) | 记忆检索必须预热/缓存，不能等 LLM 全量推理 |
| 社交打断风险 | InterruptionCost 在 Ranker 中权重极高 |
| 多人对话混淆 | 强依赖说话人分离质量 |
| 录音他人隐私 | 不存储原始音频，仅保留结构化摘要 |

#### 参考研究：LlamaPIE [S027]
- 核心创新：判断"用户何时需要帮助"的时机检测
- 输出限制：1-3 个词的极短提示
- Context 策略：从背景对话中提取关键信息，而非等用户主动询问

### 案例 3：AI Pin / 录音挂坠 / 随身记录器

#### 设备定位
全天候个人记忆外脑，通过持续音频记录和智能结构化构建用户的外部记忆系统。

#### 完整 Context 链路

```
┌─ 感知输入 ─────────────────────────────────────────────────────┐
│  麦克风(持续) + GPS/Wi-Fi + 加速度计 + 按钮交互                │
│  → 全天候音频流 (16kHz, ~115 MB/小时)                          │
│  → VAD 过滤静默段                                              │
│  → 场景分类 (会议/对话/独处/通勤)                              │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ Context 化 ──────────────────────────────────────────────────┐
│  连续音频 → 事件切分 (场景变化/对话开始结束)                    │
│  每个事件 → ASR转录 + 说话人标签 + 场景标签                    │
│  重要性评估: 有承诺? 有决策? 有新信息? 有新人?                  │
│  大量事件被过滤(static/noise/repetitive → 丢弃)               │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 记忆写入 ────────────────────────────────────────────────────┐
│  L1: 原始音频环形缓存 (最近 1 小时，本地加密)                  │
│  L4: 重要事件 → Episodic Memory (结构化摘要)                   │
│      例: {type:"commitment", content:"周五前发报告给王总"}      │
│  L5: 定期归纳 → 人物更新 / 话题知识 / 地点标记                 │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 检索召回 ────────────────────────────────────────────────────┐
│  触发: 用户主动查询 ("上周和谁讨论过预算?")                    │
│  检索: 时间范围 + 话题语义 + 人物关联                          │
│  或触发: 定时推送 (每日摘要 / 待办提醒)                        │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ Context 组装 (Compiler) ─────────────────────────────────────┐
│  Budget: ~8K tokens (查询型，可较长)                           │
│  结构: System + Query + 检索结果(按时间排序) + 输出格式         │
│  Ranker 侧重: Recency(0.20), TaskRelevance(0.35)              │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 模型决策 ────────────────────────────────────────────────────┐
│  生成: 结构化回答 / 每日摘要 / 待办清单                        │
│  格式: 时间线式回答、要点列表                                  │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 设备输出 ────────────────────────────────────────────────────┐
│  手机 App 推送 (事后异步，非实时)                              │
│  或语音播报 (用户主动触发时)                                    │
│  延迟容忍度高: <2-5s 可接受                                    │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 反馈 → 记忆更新 ────────────────────────────────────────────┐
│  用户确认摘要准确 → 强化记忆置信度                             │
│  用户编辑/纠正 → 更新记忆内容 + 记录纠正                      │
│  用户删除 → 删除记忆 + 学习"不该记这类事"                     │
└───────────────────────────────────────────────────────────────┘
```

#### 独有挑战

| 挑战 | Context 管理策略 |
|------|-----------------|
| 全天数据量巨大 (~1GB/天音频) | 激进过滤：仅 <5% 事件写入 L4 |
| 旁人隐私法律风险 | 某些地区需双方同意录音；默认不存他人声纹 |
| "该记什么"的判断 | 承诺/决策/新信息 > 日常闲聊，importance 模型关键 |
| 长期存储成本 | 分层存储：热(端侧7天) → 温(云端90天) → 冷(摘要永久) |
| 用户信任 | 必须极度透明：每日新记忆摘要推送，用户一键删除 |

#### 参考产品：Humane AI Pin [S040] / Friend Pendant [S041]
- Humane AI Pin 教训：延迟太高、准确率不够、场景理解弱 → Context 管理问题的集中体现
- Friend/Omi 方向：聚焦"记忆助手"而非全能助手，更窄的 scope → 更可控的 context

### 案例 4：桌面机器人 / 家庭机器人 / VLA 机器人

#### 设备定位
具身智能体，通过空间感知、动作执行和持续学习在物理世界执行任务。

#### 完整 Context 链路

```
┌─ 感知输入 ─────────────────────────────────────────────────────┐
│  RGB-D 摄像头 + 麦克风 + 关节编码器 + 力/触觉传感器 + LiDAR   │
│  → 环境状态: 物体位置、空间结构、障碍物                        │
│  → 用户指令: ASR + 意图理解                                    │
│  → 自身状态: 关节角度、末端位姿、载荷                          │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ Context 化 ──────────────────────────────────────────────────┐
│  视觉流 → 场景图(物体+位置+关系) (高频更新)                    │
│  用户语音 → 任务指令结构化                                      │
│  关节状态 → 运动学状态向量                                      │
│  切分: 任务阶段(导航/抓取/放置) / 环境变化 / 任务切换          │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 记忆写入 ────────────────────────────────────────────────────┐
│  L2: 环境实时状态 (物体位姿、障碍物、人的位置)                 │
│  L3: 当前任务计划 + 子任务进度                                  │
│  L4: 任务执行记录 (成功/失败 + 原因)                           │
│  L5: 物体知识 (材质/重量/抓取策略) + 空间地图                  │
│  L6: 动作规则 (Procedural): "陶瓷杯→顶部包围抓取"            │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 检索召回 ────────────────────────────────────────────────────┐
│  High-level (LLM): 用户偏好 + 任务经验 + 物体知识             │
│  Low-level (VLA): 目标物体抓取策略 + 已知失败模式              │
│  Ranker 侧重: Actionability(0.25), Confidence(0.15)           │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ Context 组装 (双层 Compiler) ────────────────────────────────┐
│                                                                │
│  ┌─ High-level (LLM, 低频 ~1次/任务) ──────────────────────┐ │
│  │  Budget: ~8K tokens                                       │ │
│  │  内容: 用户请求 + 环境语义 + 偏好 + 任务分解              │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                │
│  ┌─ Low-level (VLA, 高频 ~10Hz) ────────────────────────────┐ │
│  │  Budget: ~1K tokens + 视觉embedding                       │ │
│  │  内容: 当前观察 + 子任务 + 关节状态 + 动作约束            │ │
│  └───────────────────────────────────────────────────────────┘ │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 模型决策 + 执行 ────────────────────────────────────────────┐
│  LLM: 输出任务计划 + 子任务序列                                │
│  VLA: 输出连续动作向量 (关节速度/位置指令)                     │
│  安全检查: 力矩限制、碰撞检测、人员安全距离                   │
└────────────────────────────────┬────────────────────────────────┘
                                 ↓
┌─ 反馈 → 记忆更新 ────────────────────────────────────────────┐
│  动作成功 → 强化该物体的抓取策略 (L6 success_count++)          │
│  动作失败 → 反思失败原因 → 写入 Procedural (避免重复失败)     │
│  用户纠正 → "不是那个杯子" → 更新物体识别记忆                 │
│  用户偏好 → "轻一点" → 更新力量参数 (L5/L6)                   │
└───────────────────────────────────────────────────────────────┘
```

#### 独有挑战

| 挑战 | Context 管理策略 |
|------|-----------------|
| 物理安全 (碰撞/夹伤) | Safety 约束硬编码在 VLA context 最高优先级 |
| 双频率需求 (规划1Hz vs 动作10Hz) | 双层 Compiler：LLM低频规划 + VLA高频控制 |
| 动作不可逆性 | 每步执行前检查是否满足 Procedural Memory 中的安全规则 |
| 环境持续变化 | L2 高频更新 + 每步重新评估环境是否匹配计划前提 |
| 从失败中学习 | Reflexion 机制: 失败→反思→写入 L6 Procedural |

#### 参考研究
- **OpenVLA** [S037]: 展示 VLA 的标准输入格式——图像观察 + 自然语言指令 + 动作输出
- **MemoryVLA** [S038]: 将工作记忆(当前观察)和感知认知记忆库(历史经验)分离

## 6.5 四类设备横向对比

| 维度 | AI 眼镜 | AI 耳机 | AI Pin | 机器人 |

| **主要模型** | VLM + LLM | LLM | LLM | VLA + LLM |

## 7. 对智能硬件 Context 管理的启发

| 跨设备共性 | 启示 |
|-----------|------|
| 都需要 importance 评估 | Context Compiler 的 Trigger Detection 是通用模块 |
| 都需要 interruptibility 判断 | 主动服务决策在所有设备形态中都需要 |
| 都需要隐私过滤 | 端侧 Privacy Filter 是第一步 |
| 都需要用户反馈闭环 | Feedback → Memory Update 是记忆系统的必要组件 |
| 都需要结构化输出约束 | 每种设备的输出通道有不同限制 |

| 跨设备差异 | 启示 |
|-----------|------|
| 延迟从 <100ms 到 <2s 差异巨大 | Compiler 需要按设备配置不同的检索/排序深度 |
| 输出从 1 词到完整报告 | 不能用统一 prompt 模板，需设备特化 |
| 安全从"尴尬"到"物理伤害" | Safety 约束的强度和位置需按风险等级调整 |
| 记忆重点完全不同 | Ranker 权重必须按设备类型自适应 |

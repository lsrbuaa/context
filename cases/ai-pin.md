# 案例研究：AI Pin / 录音挂坠 / 随身记录器

## 产品定位

全天候个人记忆助手，通过持续音频记录、智能过滤和结构化抽取，构建用户的外部记忆系统。

## Context 管理链路

```
┌─────────────┐     ┌──────────────┐     ┌─────────────────┐
│  全天音频流  │────→│  VAD + 场景  │────→│  事件切分        │
│  + 位置数据  │     │  分类过滤    │     │  + 重要性评估    │
└─────────────┘     └──────────────┘     └────────┬────────┘
                                                   │
                                                   ↓
                                         ┌─────────────────┐
                                         │  结构化抽取      │
                                         │  待办/承诺/人物  │
                                         └────────┬────────┘
                                                   │
                    ┌──────────────┐               │
                    │  Episodic    │←──────────────┤
                    │  Memory      │               │
                    └──────┬───────┘               ↓
                           │             ┌─────────────────┐
                           ↓             │  定期归纳        │
                    ┌──────────────┐     │  Semantic Memory │
                    │  用户查询时   │     └─────────────────┘
                    │  检索+摘要    │
                    └──────────────┘
```

## 典型场景

### 场景 A：会议摘要生成

**触发条件**: 会议结束（通过场景变化或日历事件判断）

**Context Assembly**:
- Audio Events: 整场会议的切分片段
- Speaker Identification: 各参与者发言
- Calendar Context: 会议主题和参与者

**输出**: 结构化会议摘要
```
会议：产品周会 (14:00-15:00)
参与者：张工、李总、王设计
决策：
  - 下周三前完成原型
  - API 对接方案采用方案 B
待办：
  - [我] 周五前发送设计文档
  - [张工] 准备技术评审
```

### 场景 B：承诺追踪

**触发条件**: 检测到用户做出承诺（"我明天给你发"/"下周我来处理"）

**Context Assembly**:
- Audio: 用户说 "我周五前把报告发给你"
- Context: 与王总对话，关于 Q3 规划

**写入记忆**:
```json
{
  "type": "commitment",
  "content": "发送 Q3 规划报告给王总",
  "deadline": "2026-06-27",
  "context_person": "王总",
  "source_event": "午餐对话"
}
```

### 场景 C：生活日志查询

**触发条件**: 用户主动询问 "上周我和谁讨论过预算的事？"

**Context Assembly**:
- Query: 时间范围 (上周) + 话题 (预算) + 人物 (未知)
- Retrieval: 搜索 episodic memory
- Result: 周二与财务李姐，周四与项目组

**输出**: "上周有两次：周二和财务李姐讨论了部门预算分配，周四项目组会议上提到了 Q3 预算调整。"

## 数据过滤策略

### 什么值得记住

| 类型 | 示例 | 优先级 |
|------|------|--------|
| 承诺/待办 | "我明天发给你" | 高 |
| 决策 | "我们用方案 B" | 高 |
| 重要对话 | 与关键人物的实质讨论 | 中 |
| 新信息 | 学到的新知识 | 中 |
| 日常互动 | 普通寒暄 | 低 |
| 环境噪声 | 背景音乐、交通声 | 丢弃 |

### 什么应该丢弃

- 连续静默 > 30 秒
- 纯环境噪声
- 重复内容（同一对话的重复表述）
- 隐私敏感内容（根据用户设置）

## 存储架构

```
┌─────────────────────────────────────────┐
│  Hot Storage (端侧, 24h)                 │
│  原始音频片段，用于即时回放              │
├─────────────────────────────────────────┤
│  Warm Storage (端侧/云端, 30 天)         │
│  事件摘要，结构化抽取结果                │
├─────────────────────────────────────────┤
│  Cold Storage (云端, 永久)               │
│  Semantic Memory: 人物、偏好、知识       │
│  重要 Episodic: 关键事件                 │
└─────────────────────────────────────────┘
```

## 隐私治理

| 风险 | 缓解措施 |
|------|----------|
| 录制他人对话 | 默认仅保留摘要，不存原始音频 |
| 敏感对话泄露 | 本地处理优先，加密存储 |
| 他人隐私权 | 检测多人场景时提高过滤阈值 |
| 法律合规 | 某些地区需双方同意录音 |
| 数据泄露 | 端侧加密，最小化云端存储 |

## 关键技术挑战

1. **全天续航**: 持续录音的功耗管理
2. **实时过滤**: 在端侧快速判断重要性
3. **事件边界**: 准确切分连续音频流为有意义事件
4. **归纳能力**: 从多个 episode 中自动提炼 pattern
5. **检索精度**: 用户模糊查询的语义匹配

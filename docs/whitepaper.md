# 跨设备用户记忆底座白皮书

## 摘要

智能硬件厂商的个性化 Agent 能力，不应建立在“把更多用户数据塞进模型上下文”上，而应建立在一套跨设备用户记忆底座上。这个底座把手机、可穿戴、耳机、音箱、智能家居、车机、客服和用户显式反馈中的连续信号，转化为可解释、可更新、可检索、可遗忘、可授权使用的记忆对象，并在每次模型调用前生成高质量 Context Package。

这个方向的工程价值在于：

- 把多端数据优势转化为 Agent 可用的长期记忆能力。
- 把隐私、授权、审计和删除做成底座能力，而不是后补合规项。
- 把个性化体验从 prompt 调参推进到可评估的系统工程。
- 把智能硬件生态的设备状态、用户偏好和服务闭环连接成公共能力。

## 1. 问题定义

跨设备用户记忆底座要解决的不是“存储更多历史”，而是“在正确场景下想起正确的信息”。

普通 RAG 解决的是外部知识召回；长上下文解决的是更大输入窗口；用户记忆底座解决的是长期个人化状态管理。它必须同时处理：

- 数据来源多：设备、App、云端、客服、用户反馈。
- 数据粒度杂：原始传感器、设备日志、一次性交互、长期偏好、权限策略。
- 数据风险高：位置、健康、家庭成员、儿童、私密对话等高敏信息。
- 数据状态会变化：偏好会过期，用户会纠正，场景会冲突。
- 模型调用有预算：上下文必须短、准、结构清晰。

因此，底座的核心输出不是一个历史数据库，而是面向模型调用的 Context Package：

```text
当前任务
+ 当前设备与环境状态
+ 少量相关用户记忆
+ 证据、时间与置信度
+ 隐私、授权和行为约束
+ 可调用工具状态
= 本次模型调用上下文
```

## 2. 行业判断

大模型基础能力会持续提升，但模型能力的通用化并不会自动带来用户级差异化。智能硬件厂商真正有壁垒的部分，是持续、真实、跨场景、贴身的数据入口，以及对设备执行链路的控制。

这带来一个关键判断：

> 模型厂商更擅长提供通用推理能力，智能硬件厂商更有机会建设用户状态和设备状态的长期记忆底座。

跨设备用户记忆底座的价值不是替代模型，而是在模型调用前完成上下文工程：

- 帮模型过滤无关信息。
- 帮模型识别当前场景和权限边界。
- 帮模型稳定使用长期偏好。
- 帮模型避免过期、冲突或敏感记忆。
- 帮系统形成可审计、可评估的个性化能力。

## 3. 总体架构

跨设备用户记忆底座可以分为八层：

| 层级 | 作用 | 关键工程问题 |
|---|---|---|
| 设备与应用信号层 | 接入多端原始信号 | 哪些数据可靠、授权、可采集 |
| 数据治理层 | 去噪、去重、脱敏、最小化 | 哪些字段不能进入记忆系统 |
| 事件标准层 | 统一 Event 表达 | 多设备语义如何对齐 |
| 记忆生成层 | 从事件中抽取候选记忆 | 什么值得长期记住 |
| 记忆管理层 | 存储、更新、降权、遗忘 | 记忆如何生命周期治理 |
| 权限策略层 | 控制使用场景和可见范围 | 当前场景是否允许使用 |
| 检索组装层 | 召回、重排、压缩上下文 | 如何少取、准取、可解释地取 |
| 评估反馈层 | 衡量效果和风险 | 个性化是否真的提升能力 |

系统主链路如下：

```text
Raw Signal
 -> Event
 -> Episode
 -> Memory / Profile / Policy
 -> Retrieval & Rerank
 -> Context Package
 -> Agent Action
 -> Feedback / Audit / Forgetting
```

其中，Policy 不是附属字段，而是和 Memory 同级的底座对象。没有策略约束的用户记忆，会把个性化能力变成隐私风险。

## 4. 核心模块

| 模块 | 输入 | 输出 | 决策重点 |
|---|---|---|---|
| Event Normalizer | 原始设备日志、App 事件、用户反馈 | 标准 Event | 时间、设备、用户身份和场景是否对齐 |
| Memory Extractor | Event / Episode | 候选 Memory | 是否具备长期价值，是否来自显式反馈 |
| Memory Scorer | 候选 Memory、历史证据 | 分数和状态 | 频率、新近性、稳定性、重要度、风险 |
| Memory Store | Memory、Evidence、Policy | 可检索记忆库 | 支持检索、审计、删除和冲突关系 |
| Policy Engine | 场景、授权、敏感等级 | 使用许可 | 是否能进入模型上下文 |
| Retriever / Reranker | 当前任务、场景、记忆库 | 相关记忆候选 | 相关性、置信度、冲突、token 预算 |
| Context Composer | 任务、状态、记忆、策略 | Context Package | 顺序、格式、压缩、约束保留 |
| Feedback Updater | 用户反馈、任务结果、审计日志 | 更新 / 降权 / 遗忘 | 新信息是否覆盖旧记忆 |

## 5. 数据生命周期

跨设备记忆的生命周期必须从采集开始设计，而不是等到模型调用时再补救。

| 阶段 | 系统动作 | 关键约束 |
|---|---|---|
| Observe | 从设备和 App 观察信号 | 必须有合法来源和授权状态 |
| Normalize | 转换为标准 Event | 统一时间、身份、设备、场景 |
| Minimize | 清洗、脱敏、裁剪字段 | 不保留未来任务用不到的信息 |
| Extract | 抽取候选记忆 | 单次偶发事件默认不进入长期记忆 |
| Confirm | 评分、冲突检测、必要时待确认 | 用户显式反馈优先 |
| Retrieve | 按任务和场景召回 | 权限过滤先于模型输入 |
| Compose | 生成 Context Package | 短、准、结构化，保留约束 |
| Act | Agent 回复或调用设备工具 | 高风险动作需要确认 |
| Update | 根据结果更新、降权或遗忘 | 删除必须可执行、可审计 |

## 6. 隐私与治理

用户记忆底座的默认策略应是保守的：

- 高敏数据默认不进入模型上下文。
- 能在端侧完成的敏感摘要，优先在端侧完成。
- 每条重要记忆必须有证据来源。
- 每条记忆必须可解释、可编辑、可删除。
- 多人家庭场景必须区分身份、权限和可见范围。
- 模型侧只接收当前任务必要的信息，不接收原始历史。

### 6.1 使用决策矩阵

| 数据类型 | 默认处理 | 进入模型条件 |
|---|---|---|
| 设备偏好、亮度、音量 | 可沉淀为低敏记忆 | 任务相关即可使用 |
| 位置、作息、家庭场景 | 需要场景授权和最小化 | 明确场景需要且用户授权 |
| 健康、身份、儿童、私密对话 | 默认不进入模型 | 显式授权、必要性证明、严格审计 |
| 用户拒绝或法律禁止的数据 | 不采集、不推断、不使用 | 不允许进入 |

### 6.2 删除与遗忘

遗忘不是简单地把某条自然语言记录删除。系统还必须处理：

- 结构化字段。
- 向量索引。
- 摘要缓存。
- 评估样本。
- 审计日志中的可识别内容。
- 冲突记忆中的反向引用。

用户删除的记忆不得通过摘要、向量或缓存被再次恢复。

## 7. 评估体系

跨设备用户记忆底座至少需要评估四类能力：

| 评估类别 | 指标 | 含义 |
|---|---|---|
| 检索质量 | Memory Precision / Recall | 取出的记忆是否相关、关键记忆是否漏掉 |
| 个性化收益 | Personalization Gain | 相比无记忆版本是否提升任务成功率 |
| 上下文效率 | Context Efficiency | 每 1K token 带来的效果提升 |
| 风险治理 | Privacy Leakage / Staleness Error | 是否误用敏感或过期记忆 |
| 冲突处理 | Conflict Resolution Accuracy | 新旧偏好冲突时是否新近明确表达优先 |
| 用户控制 | Correction Rate / Deletion Success | 用户能否低成本纠错和删除 |

评估不应只看回答是否自然，而应比较至少五组基线：

1. 无用户记忆。
2. 最近 N 条对话。
3. 普通向量 RAG。
4. 混合检索 + 结构化记忆。
5. 混合检索 + 结构化记忆 + Policy + Context Composer。

## 8. 落地路线

白皮书阶段不写代码，但需要把后续工程落点定义清楚。

### 阶段 1：统一语言

- 建立 Event、Episode、Memory、Profile、Policy、Context Package 的术语表。
- 明确哪些对象属于底座，哪些属于上层 Agent 应用。
- 输出 taxonomy 和决策矩阵。

### 阶段 2：场景资产地图

- 按设备和场景列出可用信号。
- 标注可靠性、敏感度、授权状态。
- 选出最适合 PoC 的低风险场景。

### 阶段 3：评估先行

- 先设计 eval case，再设计实现。
- 每个 eval case 都包含任务、可用记忆、禁止使用记忆、期望行为和风险检查。
- 用评估结果反推记忆抽取和 Context Composer 的设计。

### 阶段 4：最小工程闭环

后续进入 PoC 时，只需要先实现四类对象：

- Event：标准化输入。
- Memory：结构化长期记忆。
- Policy：使用约束。
- Context Package：模型调用输入。

这四类对象跑通后，再扩展检索、重排、压缩、审计和用户可控界面。

## 9. 参考研究

- [A Survey of Context Engineering for Large Language Models](https://arxiv.org/abs/2507.13334)：把 Context Engineering 归纳为 context retrieval / generation、context processing、context management，并讨论 RAG、memory systems、tool-integrated reasoning 等系统实现。
- [A Survey on the Memory Mechanism of Large Language Model based Agents](https://arxiv.org/abs/2404.13501)：系统整理 LLM Agent 记忆模块的设计、更新、评估和应用。
- [A Survey of Personalized Large Language Models: Progress and Future Directions](https://arxiv.org/abs/2502.11528)：从 personalized context、personalized adapters、personalized preferences 三个层面整理个性化大模型。
- [Privacy-Preserving Retrieval-Augmented Generation with Differential Privacy](https://arxiv.org/abs/2412.04697)：指出 RAG 在敏感外部数据场景中的泄露风险，并讨论 differential privacy 下的生成策略。
